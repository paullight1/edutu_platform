System Architecture

Runtime View

Edutu has several deployable surfaces that share Supabase data and Clerk identity. The NestJS API is the main backend boundary for business logic, while some client code and Supabase edge functions still talk to Supabase directly for auth, RLS-bound reads, chat proxying, webhooks, and mobile-specific functions.

NestJS API React Vite Admin React Vite PWA Expo Mobile Next Waitlist Crawl4AI Scraper Supabase Edge Functions

Primary Boundary

The intended boundary is client to NestJS API to Supabase. The actual codebase is transitional: the web app and mobile app use backend APIs for many features, but they also instantiate Supabase clients. Admin auth currently checks Supabase directly. Mobile uses Clerk tokens with Supabase access token bridging.

Key correction: the active NestJS API is under edutu-platform/backend/services/services/api, not the top-level backend folder alone. 
Browser admin and web app
  -> Clerk or Supabase auth
  -> NestJS API for goals, opportunities, roadmaps, AI, billing, notifications, scraper, mobile control
  -> Supabase directly for selected RLS data/functions

Expo mobile
  -> Clerk Expo auth
  -> API client for backend endpoints
  -> Supabase client with Clerk token bridge for RLS and edge functions
  -> native integrations: push, widgets, haptics, offline cache

Backend API
  -> Drizzle and pg pool for core tables
  -> Supabase service-role client for admin-style data operations and scraper storage
  -> Gemini/OpenRouter adapter layer for AI features
  -> Paystack webhook and billing status

Scraper
  -> Crawl4AI and Python extraction pipeline
  -> Supabase service-role save path
  -> Admin dashboard trigger path through NestJS scraper module

Platform Interactions

User App

Users browse opportunities, create goals, follow roadmaps, save/apply to opportunities, manage profile, access premium gates, and use AI-assisted CV or roadmap tools.

  • Web routes under /app/*
  • Mobile routes under app/(app)
  • Clerk is the primary user identity layer

Admin

Admins manage dashboards, users, creators, roadmaps, blog, scraper, opportunities, settings, and mobile control-plane content.

  • Standalone admin: edutu-platform/admin
  • Embedded app admin: edutu-web-app/src/admin
  • Backend admin gates use AdminGuard

Data/AI Automation

Scraper jobs gather opportunities, AI extracts/enhances structured data, and recommendation services rank opportunities against user preferences and signals.

  • Scraper routes: /api/scraper/*
  • AI features: chat, scraper, opportunities, cv, roadmaps, quiz
  • Usage logs persist to AI tables
1
Authentication. Clerk validates user sessions for web and mobile. Backend ClerkAuthGuard accepts Clerk bearer tokens and can fall back to Supabase tokens where configured.
2
User identity normalization. Non-UUID auth IDs are converted into stable UUID-shaped IDs with toDatabaseUserId before profile lookups and Drizzle queries.
3
Feature call. Client calls a service module, which either requests the NestJS API through VITE_API_URL/EXPO_PUBLIC_API_URL or invokes Supabase directly for specific data/function paths.
4
Business logic. NestJS modules enforce feature behavior, call Drizzle/Postgres, Supabase service role, AI adapters, or provider webhooks as needed.
5
Feedback loop. Signals, progress, notifications, usage logs, scraper jobs, and admin decisions feed back into personalization and operational dashboards.

Core Process Documentation

Opportunity Discovery Process

  1. Scraper source is selected from admin or CLI.
  2. List pages are fetched with browser-like headers, delays, and page caps.
  3. Detail pages are enriched for direct apply links, images, descriptions, requirements, benefits, and deadlines.
  4. AI route scraper.extract or opportunity enhancement routes can normalize unstructured text.
  5. Records are quality scored and saved as opportunities through Supabase or Drizzle-backed paths.
  6. Admins review pending opportunities and approve, reject, edit, or bulk import.

Recommendation Process

  1. User preferences, profile attributes, interests, skills, and signals are collected.
  2. Opportunity context is normalized from title, organization, category, location, requirements, benefits, tags, and roadmap content.
  3. Rule-based matching and AI reranking produce match reasons, risks, and tags.
  4. Signals such as views, saves, applies, and dismissals are written to user_opportunity_signals.
  5. Recommendations are served back to web and mobile clients.

Roadmap Process

  1. User creates a goal or adopts/enrolls in a roadmap.
  2. Roadmap templates or creator packages define milestones and task structure.
  3. AI assist routes can ask intent questions, derive tags, and match roadmaps.
  4. Progress and feedback are stored for dashboards and recommendations.
  5. Calendar views expose enrollment deadlines and roadmap schedules.

Creator Marketplace Process

  1. User applies to become a creator with display name, bio, content type, experience, and samples.
  2. Admin reviews creator applications and updates status.
  3. Approved creators can create marketplace listings, packages, or roadmaps.
  4. Users enroll in marketplace content, spending credits or passing premium gates when applicable.
  5. Marketplace performance feeds dashboards through enrollments, tickets, and analytics.

AI Configuration Process

  1. Admin stores provider keys using the AI controller.
  2. AiEncryptionService encrypts keys and stores previews.
  3. Feature routes map to provider, model, temperature, JSON mode, fallback, and enablement state.
  4. AiService resolves defaults or saved routes at runtime.
  5. Each generation attempt logs provider, model, feature, duration, and result status.

Mobile Control Process

  1. Admin creates campaigns, feature flags, and widget feeds.
  2. Mobile app requests config via backend/mobile-control paths.
  3. MobileCampaignHost renders active campaign content in the Expo app shell.
  4. Mobile feature flags toggle product behavior without a release.
  5. Campaign events record impressions and interactions for later analysis.

Risks and Open Questions

Risk Why it matters Recommended action
High Split data access paths Some clients use NestJS, some use Supabase directly. Authorization and validation can drift. Write a clear exception list. Move sensitive writes behind NestJS or audited edge functions.
High Multiple schema homes Drizzle schema, web app Supabase migrations, mobile migrations, and shared migrations may diverge. Define a single source of truth and migration ownership policy.
Review Admin auth inconsistency Standalone admin checks Supabase profiles directly while backend uses Clerk/Supabase token guards. Document the intended admin identity provider and align all admin surfaces.
Review Service-role Supabase use Backend and scraper service-role operations bypass RLS and require careful validation. Add DTO validation, audit logs, and explicit service-role call wrappers.
Review AI key and usage governance AI routes are flexible and powerful. Misconfiguration can affect cost, privacy, and behavior. Add feature owner, cost caps, redaction, and per-route test prompts.
Review Scraper quality threshold Low-quality or duplicated opportunities degrade trust and recommendation quality. Track quality score, source reliability, dedupe decisions, and admin review outcomes.
Review Mobile control blast radius Campaigns, feature flags, and widget feeds can alter native UX remotely. Add preview, targeting, kill switch, and event audit trail for every remote config change.

Review Notes

What Looks Strong

  • The backend is modular and feature-oriented.
  • AI routing is centralized rather than hardcoded per feature.
  • Mobile has thoughtful infrastructure for offline cache, native shell behavior, push, widgets, and remote campaigns.
  • The scraper has a clear extraction, cleaning, and persistence shape.
  • The product has a coherent opportunity to roadmap to marketplace loop.

What Needs Discipline

  • Data access boundaries need to be explicit and enforced.
  • Schema ownership should be consolidated or documented tightly.
  • Admin and service-role operations need auditability.
  • AI features need cost, privacy, and fallback runbooks.
  • Multiple app surfaces need shared route, env, and deployment documentation.
