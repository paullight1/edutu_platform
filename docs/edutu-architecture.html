Executive Summary

What Edutu Is

Edutu is a multi-surface opportunity platform for scholarships, roadmaps, creator marketplace, AI assistance, CV tooling, notifications, billing, and mobile growth controls.

Core Architecture

The platform is organized around a NestJS API, Supabase Postgres, React/Vite web apps, an Expo mobile app, Supabase edge functions, and a Crawl4AI scholarship scraper.

Main Rule

Client features should prefer the backend API for business logic. Some current surfaces still call Supabase directly for auth, RLS-backed data access, edge functions, or admin reads.

Repository boundary correction: this is intentionally not a monorepo. edutu-platform/ and edutumobile/ should be treated as separate repository boundaries that happen to be present under one local workspace folder for this scan. In this workspace copy, Git metadata was not exposed, so the documentation records the intended ownership boundaries rather than relying on .git discovery.

System Architecture

Edutu is best understood as a hub-and-spoke system. Product clients render user and admin workflows. The NestJS API owns high-trust business logic and AI routing. Supabase stores product data and also hosts edge functions for selected serverless workflows.

Clients
Main Web App-> Admin Dashboard-> Expo Mobile-> Waitlist Site
Service Layer
NestJS API-> Supabase Edge Functions-> Crawl4AI Scraper
Data and Identity
Supabase Postgres-> Clerk Auth-> Supabase Auth/RLS Bridge
External Services
Gemini-> OpenRouter-> Paystack-> RevenueCat-> Apify Actors-> n8n
The strongest boundary is the backend API. The weakest boundary is the current split between direct Supabase client access and backend mediated access. That split should be documented per feature and tightened where business rules require server enforcement.

Runtime Surfaces

Main Web App

Path: edutu-platform/edutu-web-app

Routes are declared in src/App.tsx. Public routes include landing, about, opportunities, blog, mentor, auth, and billing. Authenticated app routes are nested under /app and include home, opportunities, community, achievements, saved/applied items, goals, opportunity roadmaps, packages, settings, CV, personalization, and creator tools.

The web app uses Clerk for identity, Supabase for selected data access and RLS bridging, Vite for builds, and Capacitor for the Android wrapper.

Standalone Admin

Path: edutu-platform/admin

Routes in src/App.tsx cover dashboard, opportunities, users, creators, roadmaps, blog, settings, scraper (/edutu-engine), mobile control, and profile. Auth is Supabase session based, with admin access determined by profiles.role === "admin".

Note that the AGENTS guide mentions /scraper, while this route file currently exposes the scraper page at /edutu-engine.

Expo Mobile App

Path: edutumobile

The mobile root layout wraps the app in Clerk, theme, offline, gesture, error boundary, status bar, offline banner, and mobile campaign host providers. It bridges Clerk tokens into Supabase through setSupabaseAccessTokenGetter.

Route groups include (auth), (app), and admin. The main app layout adds tab navigation, voice entry, push notifications, haptics, and mobile campaign UI.

Waitlist Site

Path: edutu-platform/edutu-web

The Next.js site renders a public waitlist landing page with a typed hero message and WaitlistForm. It includes SQL scripts for waitlist table setup and an API route at app/api/waitlist/route.ts.

Backend API

The NestJS API is imported through AppModule, configured with global config, schedule support, throttling, and the auth guard. CORS allows localhost web/admin/mobile origins plus ADMIN_URL and MOBILE_APP_URL.

Data Model

Drizzle schema definitions live in edutu-platform/backend/services/services/api/src/db/schema.ts. The API uses DATABASE_URL through a node-postgres pool. Several modules also use a Supabase service-role client for direct table access when the feature expects Supabase response shapes or RLS helper behavior.

Identity and Profiles

  • profiles stores app-owned user profile data and role.
  • Clerk IDs are converted to database-safe IDs before lookup.
  • Admin access can come from profiles.role or configured admin emails.

Opportunity Graph

  • opportunities stores public opportunity records.
  • user_opportunity_preferences stores personalization preferences.
  • user_opportunity_signals records clicks, saves, views, and feedback signals.

Learning and Roadmaps

  • goals and milestones model user plans.
  • roadmaps, enrollments, intents, and feedback model reusable learning paths.
  • quizzes and flashcard tables support study workflows.

Monetization and Control

  • transactions, marketplace listings, packages, and enrollments support creator and billing flows.
  • mobile_app_campaigns, mobile_feature_flags, and widget_feeds power mobile control plane behavior.
  • ai_provider_keys, ai_routes, and ai_usage_logs support AI governance.

Core Processes

1. Authentication and Request Authorization

  1. Client authenticates with Clerk or Supabase depending on surface.
  2. Client sends Authorization: Bearer <token> to protected backend endpoints.
  3. ClerkAuthGuard skips routes marked public.
  4. The guard verifies Clerk tokens when CLERK_SECRET_KEY is available.
  5. If Clerk verification fails, it tries supabase.auth.getUser(token).
  6. The guard attaches request.user with user ID, auth ID, email, role, and provider.
  7. Admin-only features use AdminGuard or route-level admin logic.

2. Opportunity Discovery

  1. User opens opportunities from web or mobile.
  2. Client fetches active opportunities from backend or Supabase-backed service layer.
  3. Personalization can combine user profile, preferences, signals, and ranking service output.
  4. Opportunity detail routes record analytics and can start roadmap generation or goal adoption.
  5. Saved and applied lists are surfaced through user-specific services.

3. Scraper Ingestion

  1. Admin triggers POST /api/scraper/run or scheduler executes a configured cron.
  2. Scraper service loads scraper settings and enabled sources from Supabase tables.
  3. List pages are fetched with browser-like headers and conservative delays.
  4. Detail pages are enriched with Cheerio extraction and AI-assisted JSON extraction.
  5. Records are normalized, quality checked, and saved as opportunities.
  6. Stats and job history are shown in the admin scraper surface.

4. AI Operations

  1. Feature service calls AiService.generateText or generateJson.
  2. Feature route is resolved from database config or default route map.
  3. Provider adapters execute against Gemini or OpenRouter.
  4. JSON responses are normalized and parsed for structured workflows.
  5. Usage is logged with latency and error metadata.

5. Mobile Control Plane

  1. Mobile app loads remote config through /mobile-control/config.
  2. Backend returns active campaigns, enabled feature flags, widget feeds, and server time.
  3. Mobile renders campaigns through MobileCampaignHost.
  4. Mobile records campaign events through /mobile-control/events.
  5. Admin manages campaigns, flags, and widget feeds through standalone admin or backend admin endpoints.

6. Notification Process

  1. Users register push tokens from mobile.
  2. Notification preferences determine alert categories and quiet hours.
  3. Admin or system events create notifications or queued broadcasts.
  4. Due notifications are processed through admin queue endpoint or scheduled operations.
  5. Users can mark individual notifications or all notifications as read.

Security and Auth Notes

Current Controls

  • Backend has a global auth guard with public-route opt-out.
  • Backend throttling is configured globally at 100 requests per minute.
  • Admin guard checks configured admin emails and user.role === "admin".
  • AI provider keys are encrypted before storage and exposed only as previews.
  • Supabase service-role clients are server-side only.

Review Concerns

  • Admin and client surfaces still include direct Supabase access paths that need explicit ownership rules.
  • The standalone admin uses Supabase auth, while main web/mobile use Clerk first. This creates identity consistency risk.
  • API docs should identify which endpoints are public, authenticated, admin-only, and webhook-only.
  • Webhook verification details should be audited before production billing rollout.

Quality Review

Strengths

  • Feature-based NestJS modules make backend ownership clear.
  • Drizzle schema centralizes many core table definitions.
  • AI routing is configurable and usage-aware.
  • Mobile has explicit offline, token bridge, notification, and campaign host layers.
  • The scraper has separate extraction, cleaning, persistence, CLI, and backend control surfaces.

Risks

  • Multiple Supabase folders and migration streams can diverge.
  • The direct-Supabase versus backend-API rule is not consistently enforced in clients.
  • The standalone admin and main app admin are separate surfaces with overlapping responsibilities.
  • Route labels are inconsistent: scraper is documented as /scraper but standalone admin uses /edutu-engine.
  • Installed node_modules directories inside the local workspace increase scan noise and can hide real structure.
  • Because repo metadata is not visible in this workspace copy, CI/release ownership should be confirmed inside each actual repository checkout.

Recommended Architecture Decisions

  1. Declare Edutu as an intentional multi-repo system in every README and onboarding document.
  2. Declare edutu-platform/backend/services/services/api as the canonical backend root inside the platform repository documentation and scripts.
  3. Create one canonical database migration source and mark other migration folders as historical, mobile-local, or pending merge.
  4. Document feature-by-feature whether clients call backend, Supabase table APIs, or Supabase edge functions.
  5. Decide whether standalone admin or embedded /admin/* is the long-term admin surface.
  6. Add endpoint-level auth classification: public, authenticated, admin, webhook, service-only.
